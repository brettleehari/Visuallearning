03 — The Five Alice Strategies

How Apps Build Stateful Magic on Stateless APIs

Just as HTTP needed cookies, sessions, and databases — LLMs need these strategies. Click each row to expand.

Strategy Approach Trade-off
▶Full History Replay Re-send every message with each API call Simple but expensive — costs grow per turn
How it works
Every message ever exchanged is re-sent with each API call. The model sees the complete conversation verbatim.
When it breaks
When the conversation exceeds the context window limit. The API call simply fails — no graceful degradation.
Who uses it
Simple chatbot wrappers, early-stage apps, short-lived task conversations. Good for prototyping.
Book analogy
"Alice reads Dr. Neural the entire notebook before every visit."
▶Sliding Window + Summarization Keep recent messages, compress older ones Balanced — but early details become lossy
How it works
Keep the last N messages verbatim. Older messages get compressed into a summary by the LLM itself.
When it breaks
When the user references specific details from early in the conversation that were lost during summarization.
Who uses it
ChatGPT (primary strategy), Aider (for chat history management).
Book analogy
"Alice summarizes last week's notes but reads today's in full."
▶RAG — Retrieve on Demand Search for relevant docs and inject them Efficient — but retrieval quality is the bottleneck
How it works
Index your knowledge base, search for relevant chunks per-query, inject only the most relevant documents.
When it breaks
When retrieval returns irrelevant or incomplete chunks. Bad embeddings = bad context = bad answers.
Who uses it
Perplexity (web search), Cursor (codebase indexing), Claude Projects.
Book analogy
"RAG is just: file processing + database operations + API calls. You know all three."
▶Persistent Memory Layer Extract key facts into a separate database Cross-session continuity — but what to remember?
How it works
Extract key facts (name, preferences, projects) into a separate database. Inject relevant memories per-conversation.
When it breaks
When memory extraction gets it wrong — saving irrelevant facts, missing important ones, or injecting stale info.
Who uses it
ChatGPT Memory, Claude Memory. Persists across conversations.
Book analogy
"Like a good family doctor who remembers you between visits."
▶Server-Side Compaction Auto-summarize old messages at the server Seamless — but early details become lossy
How it works
The server detects when context is approaching limits and triggers a summarization pass on older messages automatically.
When it breaks
Same as summarization — early details become lossy. The user has no control over what gets compressed.
Who uses it
Claude uses server-side compaction at the 200K+ window limit. Conversation continues indefinitely.
Book analogy
"The future of AI is getting better at managing state around stateless intelligence."
04 — Deep Dive: Aider (Open Source)

How Aider Actually Manages Context

Unlike other apps where we make educated assumptions, Aider is open source — we can see exactly how it works.

AIDER CONTEXT PIPELINE — VERIFIED FROM SOURCE CODE
STEP 1

Tree-sitter AST Parsing

Uses tree-sitter to parse every file in the git repo into ASTs. Extracts definitions and references using language-specific tags.scm query files. ~60 languages supported.

STEP 2

PageRank Graph Analysis

Each file becomes a node in a directed graph. Runs NetworkX personalized PageRank with a 50x multiplier for files in the active chat. Transitive importance is captured.

STEP 3

Repo Map Generation (~1K tokens)

Top-ranked files become a concise structural map — class names, function signatures, types. Not full code, just the skeleton. Configurable via --map-tokens.

STEP 4

Context Assembly

Priority order: System prompt + Repo map + Added files (full content via /add) + Chat history (auto-summarized when over limit).

STEP 5

Chat History Summarization

ChatSummary class uses the LLM in a background thread to summarize older conversation chunks when exceeding --max-chat-history-tokens.
