01 — The Core Problem

Every API Call Ships the Entire Conversation

Because the LLM remembers nothing, every single API call must include ALL previous context. As conversations grow, this creates a fundamental tension: the context window is finite, but conversations are not.

"The AI model (Dr. Neural) is stateless and remembers nothing. The application layer (Alice) maintains conversation history and feeds it back as context, creating the illusion of memory and continuity."

03 — The Five Alice Strategies

How Apps Build Stateful Magic on Stateless APIs

Just as HTTP needed cookies, sessions, and databases to create the modern web — LLMs need these strategies to create the illusion of intelligent, continuous conversation. Click each row to expand.

Strategy Approach Trade-off
▶Full History Replay Re-send every message with each API call Simple but expensive — costs grow per turn
How it works
Every message ever exchanged is re-sent with each API call. The model sees the complete conversation verbatim, from message #1 to the latest.
When it breaks
When the conversation exceeds the context window limit. At that point, the API call simply fails — there's no graceful degradation built in.
Who uses it
Simple chatbot wrappers, early-stage apps, short-lived task conversations. Good for prototyping.
Book analogy
"Alice reads Dr. Neural the entire notebook before every visit."
▶Sliding Window + Summarization Keep recent messages, compress older ones Balanced — but early details become lossy
How it works
Keep the last N messages verbatim. Older messages get compressed into a summary by the LLM itself. The summary replaces the original messages in context.
When it breaks
When the user references specific details from early in the conversation that were lost during summarization. "What was that exact error message from earlier?"
Who uses it
ChatGPT uses this as its primary strategy. Aider uses it for chat history management with configurable token budgets.
Book analogy
"Alice summarizes last week's notes but reads today's in full."
▶RAG — Retrieve on Demand Search for relevant docs and inject them Efficient — but retrieval quality is the bottleneck
How it works
Don't pack everything in. Index your knowledge base, search for relevant chunks per-query, and inject only the most relevant documents into the context window.
When it breaks
When the retrieval system returns irrelevant or incomplete chunks. Bad embeddings = bad context = bad answers. "Garbage in, garbage out" — but for context.
Who uses it
Perplexity (web search as RAG), Cursor (codebase indexing), Claude Projects (knowledge base search).
Book analogy
"RAG is just: file processing + database operations + API calls. You know all three."
▶Persistent Memory Layer Extract key facts into a separate database Cross-session continuity — but what to remember?
How it works
Extract key facts (name, preferences, projects) into a separate database. Inject relevant memories per-conversation. The context window stays lean while the app "remembers" you.
When it breaks
When the memory extraction gets it wrong — saving irrelevant facts, missing important ones, or injecting stale information. Memory curation is an unsolved UX problem.
Who uses it
ChatGPT Memory, Claude Memory. These persist across conversations — a layer above the context window itself.
Book analogy
"Advanced RAG maintains conversation context like a good family doctor who remembers you between visits."
▶Server-Side Compaction Auto-summarize old messages at the server Seamless — but early details become lossy
How it works
The server automatically detects when the context window is approaching limits and triggers a summarization pass on older messages. The user doesn't even know it happened.
When it breaks
Same as summarization — early details become lossy. The difference is the user has no control. It's a trade-off between UX simplicity and user agency.
Who uses it
Claude uses server-side compaction when approaching the 200K+ window limit. The conversation continues indefinitely — seamlessly.
Book analogy
"The future of AI is getting better at managing state around stateless intelligence."

04b — Deep Dive: Aider (Open Source)

How Aider Actually Manages Context

Unlike the other apps where we're making educated assumptions, Aider is open source — we can see exactly how it works. Here's the real architecture.

AIDER CONTEXT PIPELINE — VERIFIED FROM SOURCE CODE
STEP 1

Tree-sitter AST Parsing

Aider uses tree-sitter to parse every file in the git repo into Abstract Syntax Trees. It extracts definitions (where functions/classes are defined) and references (where they're used) using language-specific tags.scm query files. Supports ~60 languages with full repo map support.

STEP 2

PageRank Graph Analysis

Each source file becomes a node in a directed graph with edges based on dependencies. Aider runs NetworkX's personalized PageRank with a 50x multiplier for files referenced in the active chat. Files referenced by many important files rank high — even if not directly mentioned.

STEP 3

Repo Map Generation (~1K tokens default)

The highest-ranked files get included in a concise structural map — class names, function signatures, type definitions. Not the full code, just the skeleton. Configurable via --map-tokens (default: 1,000 tokens). This gives the LLM a bird's-eye view of the entire codebase.

STEP 4

Context Assembly

The final prompt is assembled in priority order: System prompt + Repo map + Added files (full content of files you /add) + Chat history (with auto-summarization when exceeding --max-chat-history-tokens).

STEP 5

Chat History Summarization

When chat history exceeds the configured token limit, Aider's ChatSummary class uses the LLM to create a concise summary of older conversation chunks in a background thread. The summary replaces the original messages — the conversation continues without blocking.

06 — The Hidden Danger

Context Rot: When More Is Less

Performance Degrades as the Window Fills

Research shows that as context windows fill up, models lose accuracy. Below 50% full, they lose tokens in the middle. Above 50%, they lose the earliest tokens. This is why smart context management isn't optional — it's essential.

Strong recall "Lost in the Middle" zone Strong recall

This is exactly why your book's analogy of the News Editor is so apt — limited space, unlimited information. The editor must choose wisely. Statelessness isn't a limitation — it's a feature, but only if the application layer manages context intelligently.
